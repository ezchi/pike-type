# Spec 006 â€” Rename Repository to pike-type

## Overview

Rename the project from **typist** to **pike-type**. The Python package (import path) becomes `piketype`, the CLI command becomes `piketype`, and the distribution name in `pyproject.toml` becomes `pike-type`. This is a comprehensive rename touching source directories, imports, discovery conventions, generated output names, test fixtures, golden files, documentation, and the project constitution.

"Repository rename" in this spec means the project identity (package name, distribution name, CLI command, generated output branding). The local checkout directory name and Git remote URL are out of scope â€” those are manual steps the maintainer handles separately.

## Naming Convention

| Context                  | Old Name              | New Name               |
|--------------------------|-----------------------|------------------------|
| Repository               | `typist`              | `pike-type`            |
| PyPI / distribution name | `typist`              | `pike-type`            |
| Python import path       | `typist.*`            | `piketype.*`           |
| Source directory          | `src/typist/`         | `src/piketype/`        |
| CLI command              | `typist`              | `piketype`             |
| Discovery directory      | `typist/`             | `piketype/`            |
| Runtime outputs          | `typist_runtime*`     | `piketype_runtime*`    |
| Manifest file            | `typist_manifest.json`| `piketype_manifest.json`|
| Generated headers        | `Generated by typist` | `Generated by piketype`|
| C++ namespace filter     | removes `"typist"`    | removes `"piketype"`   |
| Base exception class     | `TypistError`         | `PikeTypeError`        |

## User Stories

- **US-1:** As a developer, I want the CLI command to be `piketype` so that the tool has its new brand identity.
- **US-2:** As a developer, I want `import piketype` to work so that the Python package matches the new name.
- **US-3:** As a user with existing DSL modules, I understand that my `typist/` directories must be renamed to `piketype/` after upgrading.
- **US-4:** As a contributor, I want all internal references to be consistent so that the codebase has no stale "typist" references.

## Functional Requirements

### FR-1: Rename source package directory

Rename `src/typist/` to `src/piketype/`. All Python modules under this tree retain their relative structure; only the top-level package name changes.

### FR-2: Update all internal imports

Every `from typist.*` and `import typist.*` statement across source and test files must change to `from piketype.*` / `import piketype.*`.

### FR-3: Update `pyproject.toml`

- `name` â†’ `"pike-type"`
- `[project.scripts]` â†’ `piketype = "piketype.cli:main"`
- `[tool.setuptools.packages.find]` `where` remains `["src"]` (unchanged)
- `[tool.pytest.ini_options]` `pythonpath` remains `["src"]` (unchanged)

### FR-4: Update CLI entry point

In `cli.py`, change `prog="typist"` to `prog="piketype"`.

### FR-5: Update discovery convention

In `discovery/scanner.py`:
- Rename `is_under_typist_dir` â†’ `is_under_piketype_dir`
- Change the literal string check from `"typist"` to `"piketype"`
- Rename `find_typist_modules` â†’ `find_piketype_modules`
- Update all error messages: `"is not a valid typist module"` â†’ `"is not a valid piketype module"`, `"is not under a typist/ directory"` â†’ `"is not under a piketype/ directory"`

All callers of these functions must be updated accordingly.

### FR-6: Update generated file names and content

- **Headers:** `"Generated by typist"` â†’ `"Generated by piketype"` in `backends/common/headers.py`, `backends/py/emitter.py`, `backends/runtime/emitter.py`.
- **Runtime package names:**
  - Python: `typist_runtime` â†’ `piketype_runtime`
  - SystemVerilog: `typist_runtime_pkg` â†’ `piketype_runtime_pkg`
  - C++: namespace `typist_runtime` â†’ `piketype_runtime`, header guard `TYPIST_RUNTIME_HPP` â†’ `PIKETYPE_RUNTIME_HPP`, file names `typist_runtime.*` â†’ `piketype_runtime.*`
- **Manifest:** `typist_manifest.json` â†’ `piketype_manifest.json`
- **C++ emitter namespace filter:** Change `part != "typist"` to `part != "piketype"` in `cpp/emitter.py`.

### FR-7: Update `paths.py`

All path constants referencing `typist_runtime*` or `typist_manifest*` must use `piketype_runtime*` / `piketype_manifest*`.

### FR-8: Update validation error messages

All occurrences of `"typist file defines no DSL objects"` must change to `"piketype file defines no DSL objects"`. This message appears in both `validate/engine.py` and `commands/gen.py`.

### FR-9: Rename public exception classes

In `errors.py`:
- `TypistError` â†’ `PikeTypeError`
- Update docstrings: `"Base exception for typist."` â†’ `"Base exception for piketype."`
- All subclasses (`DiscoveryError`, `ValidationError`, `EmissionError`) already inherit from the base; only the base class name changes.
- All `except TypistError` and `raise TypistError` call sites must use `PikeTypeError`.

### FR-10: Update test fixtures

All test fixture directories with `*/typist/` path segments must be renamed to `*/piketype/`:
- `tests/fixtures/*/project/alpha/typist/` â†’ `tests/fixtures/*/project/alpha/piketype/`
- `tests/fixtures/*/project/beta/typist/` â†’ `tests/fixtures/*/project/beta/piketype/`

The negative-test fixture `tests/fixtures/outside_typist/` must be renamed to `tests/fixtures/outside_piketype/`, and its internal directory `alpha/not_typist/` must be renamed to `alpha/not_piketype/`.

DSL import statements inside fixture Python files (e.g. `from typist.dsl import Const`) change to `from piketype.dsl import Const`. These are loaded at runtime by the tool, so they must use the new package name.

All `__pycache__` directories inside fixtures must be deleted (they contain stale bytecode with old module paths).

### FR-11: Update golden files

All golden output files must reflect the new naming:
- `typist_manifest.json` files renamed to `piketype_manifest.json`
- Directory paths `py/alpha/typist/` â†’ `py/alpha/piketype/` (and similarly for `sv/`, `cpp/`, `beta/`)
- Runtime files: `typist_runtime*` â†’ `piketype_runtime*`
- Content within golden files (generated headers, manifest JSON content, import paths) updated accordingly

### FR-12: Update test source files

All test files (`tests/test_*.py`) must be updated:
- Imports from `typist.*` â†’ `piketype.*`
- Fixture path references `"typist"` â†’ `"piketype"` (including `"outside_typist"` â†’ `"outside_piketype"`, `"not_typist"` â†’ `"not_piketype"`)
- CLI invocations `-m typist.cli` â†’ `-m piketype.cli`
- Test method names containing `typist` â†’ `piketype` (e.g. `run_typist` â†’ `run_piketype`, `test_generates_all_repo_typist_modules` â†’ `test_generates_all_repo_piketype_modules`)
- String literals referencing `typist` in test data (module names, namespace parts, paths)
- Expected error message text that references "typist" must be updated

### FR-13: Update documentation

All documentation files must be updated to replace "typist" references:
- `README.md`: All references to `typist` replaced with `piketype` (CLI command) or `pike-type` (project name) as contextually appropriate.
- `docs/rfc-v1.md`: Update references. This file is the `readme` in `pyproject.toml`, so it must reflect the current project name.
- `docs/architecture.md`: Update all CLI command references and paths.
- `docs/v1-product-spec.md`: Update all references.
- `docs/milestone-01-const-sv.md`: Update references.
- `docs/requirements.org`: Update any references to "typist".
- Any other file under `docs/` containing "typist" references must also be updated (e.g. `docs/planning.org`, `docs/ir-schema.md`).

All `typist_` prefixed names in documentation (e.g. `typist_manifest.json`, `typist_runtime_pkg.sv`, `typist_utils.py`) become `piketype_` prefixed (`piketype_manifest.json`, `piketype_runtime_pkg.sv`, `piketype_utils.py`). This follows the naming convention table in this spec.

### FR-14: Update project constitution

In `.steel/constitution.md`:
- Project layout section: `src/typist/` â†’ `src/piketype/`
- CLI reference: `typist gen` â†’ `piketype gen`
- Discovery references: scanning for `typist/` modules â†’ `piketype/` modules
- All `cli.py` and path references
- Any other references to the old name

### FR-15: Update `__init__.py` docstrings and version

The package `__init__.py` docstring must change from `"Typist package."` to `"Pike-type package."`.

The `__version__` field stays at its current value (`"0.1.0"`). The `pyproject.toml` `version` field stays at `"0.2.0"`. These two version numbers are intentionally independent â€” `__version__` tracks the runtime API version while `pyproject.toml` tracks the distribution version. Bumping either is out of scope for this spec.

### FR-16: Update `uv.lock`

After renaming, regenerate `uv.lock` so it reflects the new package name. This is a generated file and should be updated by running `uv lock`.

## Non-Functional Requirements

- **NFR-1: Atomic rename.** All changes land in a single coherent set of commits on one branch. The codebase must never be in a half-renamed state on `develop`.
- **NFR-2: Tests pass post-rename.** The full test suite must pass after the rename. This is the primary correctness gate.
- **NFR-3: No stale references.** After cleanup (see NFR-5), a text search for the literal string `typist` (case-insensitive) must return zero matches in `src/`, `tests/`, `pyproject.toml`, `README.md`, `docs/`, and `.steel/constitution.md`. Excluded: `.git/`, `specs/`, `uv.lock` (generated).
- **NFR-5: Clean build artifacts before validation.** Before running any stale-reference check: delete all `__pycache__/` directories under `tests/fixtures/` and `src/`, delete `src/*.egg-info/` directories. These contain cached/generated content with old names.
- **NFR-4: Deterministic output preserved.** Generated code must remain byte-for-byte reproducible â€” only the name tokens change, not structure or ordering.

## Acceptance Criteria

- **AC-1:** `pip install -e .` succeeds and creates a `piketype` CLI command.
- **AC-2:** `piketype gen <fixture>` produces correct output matching updated golden files.
- **AC-3:** All existing tests pass with `python3 -m unittest discover -s tests` (zero failures, zero errors). The test suite also passes under `python3 -m pytest` as a secondary check.
- **AC-4:** After running cleanup per NFR-5, the following command returns no matches (exit code 1):
  ```
  grep -rI -i "typist" --exclude-dir=__pycache__ --exclude-dir='*.egg-info' --exclude='*.pyc' src/ tests/ pyproject.toml README.md docs/ .steel/constitution.md
  ```
  The `-I` flag excludes binary files. The `--exclude-dir` flags are defense-in-depth after NFR-5 cleanup.
- **AC-5:** The discovery scanner correctly identifies modules under `piketype/` directories and rejects modules not under `piketype/`.
- **AC-6:** Generated file headers say "Generated by piketype".
- **AC-7:** Generated runtime files are named `piketype_runtime*`.
- **AC-8:** Manifest output is named `piketype_manifest.json`.
- **AC-9:** `basedpyright` strict mode passes with zero errors.
- **AC-10:** Generated C++ code uses namespace `piketype_runtime` and header guard `PIKETYPE_RUNTIME_HPP`.
- **AC-11:** `python3 -c "import piketype; print(piketype.__version__)"` succeeds and prints the version string.
- **AC-12:** The base exception class is `PikeTypeError` and all subclasses inherit from it.
- **AC-13:** `uv.lock` contains `name = "pike-type"` and no stale `name = "typist"` entry.
- **AC-14:** No file or directory path under `src/` or `tests/` contains the substring `typist` (verified via `find src/ tests/ -name '*typist*'` returning empty).

## Out of Scope

- **Backward compatibility aliases.** No shim mapping `typist` â†’ `piketype`. This is a clean break.
- **Migration tooling.** No script to rename users' existing `typist/` directories. This is documented as a breaking change.
- **PyPI publishing.** Publishing to PyPI under the new name is out of scope for this spec.
- **Git remote URL changes.** Renaming the GitHub repository itself (the remote) is a manual step outside this spec.
- **Version bump.** `__version__` stays at its current value. A version bump will be a separate change.
- **Local checkout directory name.** Renaming the local directory from `typist/` to `pike-type/` is a manual step for the maintainer.
