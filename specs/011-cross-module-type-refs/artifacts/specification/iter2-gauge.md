## New Issues

### BLOCKING
- **B1 — FR-12 / AC-6 specify the wrong default C++ qualified name.** The spec says default cross-module fields render as `::alpha::piketype::foo::byte_t_ct` (`spec.md:267-269`, `spec.md:339`). Source says otherwise: `_build_namespace_view` builds `qualified = "::".join(p for p in module.ref.namespace_parts if p != "piketype")` (`src/piketype/backends/cpp/view.py:241-245`), and current goldens use namespaces such as `namespace alpha::types` (`tests/goldens/gen/struct_sv_basic/cpp/alpha/piketype/types_types.hpp:13`). Fix the spec and AC-6 to expect `::alpha::foo::byte_t_ct`, or explicitly scope a namespace-policy change and accept the resulting existing-golden churn. The latter contradicts NFR-1 (`spec.md:324`).
- **B2 — FR-1's `sys.modules` restore contract is not exception-safe for pre-existing owned keys.** FR-1 records `pre_keys`, pre-cleans module names/parents, then finally pops keys not in `pre_keys` (`spec.md:140-145`). That does not restore original module objects for keys that were in `pre_keys` but were popped and reloaded during the run; the new module remains because its key is in `pre_keys`. This is not a real "restore to `pre_keys`". Fix by snapshotting original `(key, module_object)` values for every owned module/parent key, then in `finally` restore originals and remove keys that were absent before the run.
- **B3 — FR-10 / AC-4 miss cross-module synth imports in SV test packages.** Current helper field rendering emits raw scalar typedefs for scalar `TypeRefIR`s (`src/piketype/backends/sv/view.py:370-376`), and `module_test.j2` only emits the same-module synth import passed as `synth_package_import` (`src/piketype/backends/sv/templates/module_test.j2:3-4`). FR-10 only adds `import foo_test_pkg::*;` (`spec.md:242-247`), and AC-4 only checks `bar_pkg::*` plus `foo_test_pkg::*` (`spec.md:337`). Package imports are not transitive without export declarations, so `foo_test_pkg::*` does not make `foo_pkg::byte_t` available for `byte_t field1;`. Require test packages to import the cross-module synth package too, or require cross-module scalar fields in helper classes to use `byte_t_ct` and update the helper step semantics.
- **B4 — AC-23 is a false-negative grep and does not verify NFR-7.** AC-23 literally greps for `"import {pkg}_pkg"` in three `view.py` files (`spec.md:356`). That exact command returns no hits against the existing inline import string at `src/piketype/backends/sv/view.py:704`, while a broader grep for `"import .*_pkg"` does catch it. As written, an implementer can add inline f-strings like `f"import {target}_pkg::*;"` and still pass AC-23. Replace it with a concrete regex/check that catches f-strings and string concatenation in backend Python, and decide whether the existing same-module import must move into structured template data to satisfy NFR-7.
- **B5 — FR-9 dedupes SV imports by basename, which is not a module identity.** FR-9 requires one import per unique target package basename (`spec.md:231-235`). The scanner can discover two different modules with the same stem, and duplicate basename validation only runs for `--namespace` (`src/piketype/commands/gen.py:31-32`, `src/piketype/validate/namespace.py:99-119`). If `M` references `alpha.piketype.foo` and `beta.piketype.foo`, both map to `foo_pkg` and the spec loses a dependency. Add validation that cross-module SV dependency basenames are unique, or change package naming/import naming to include enough namespace to distinguish modules.

### WARNING
- **W1 — FR-8's flag-bit literal collision rule is source-false.** FR-8 says wildcard import brings enum literals and flag-bit names into package scope (`spec.md:225`). Enum literals are package identifiers in the generated typedef (`src/piketype/backends/sv/templates/_macros.j2:80-84`), but flag bits are struct members inside a typedef (`src/piketype/backends/sv/templates/_macros.j2:64-76`), not package-level imported symbols. Keep cross-imported enum literal collision validation; remove flag-bit names from that rule unless another backend-specific collision is being targeted.
- **W2 — FR-7 is concrete enough for view builders but still omits emitter-level wording.** Existing emitters call the view entry points with only `module` (`src/piketype/backends/sv/emitter.py:37-49`, `src/piketype/backends/py/emitter.py:40-44`, `src/piketype/backends/cpp/emitter.py:33-35`). FR-7 implies the new index must be threaded through those calls, but a one-line requirement to build the repo index once per emitter and pass it into each module view would remove ambiguity.

### NOTE
- **N1 — Parent namespace packages are only partially covered by FR-1.** The prefix cleanup rule (`spec.md:142`) does cover newly-created `alpha` / `alpha.piketype` namespace packages when they were absent at run start. It does not handle pre-existing parent entries correctly because of B2.
- **N2 — Commit C's existing-golden parity is plausible.** Current fixtures do not import DSL objects across piketype modules, and existing manifest goldens all have empty `dependencies`. Adding dependency serialization should not change existing fixtures if same-module references stay excluded as FR-4 requires (`spec.md:182`).
