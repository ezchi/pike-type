# Spec 011 — Cross-Module Type References

## Overview

Allow a `Struct()` member (and any future user-type member) to reference a named DSL type that is **defined in a different Python DSL module**, and have the generated SystemVerilog, C++, and Python outputs preserve that named reference and emit the appropriate cross-package import.

### The Bug This Replaces

Today, when one DSL module imports a named type from another module and uses it as a struct member, two things go wrong:

1. **Silent flattening.** The cross-module type identity is lost during freeze. The struct field is emitted as if the user had written an inline scalar with the same width.
2. **Validator never fires.** The validation rule at `src/piketype/validate/engine.py:69-73` ("cross-module type references are not supported in this milestone") is unreachable for scalar aliases because the `TypeRefIR` is never constructed in the first place.

The root cause is in the loader/freeze path:

- `src/piketype/loader/python_loader.py` calls `sys.modules.pop(module_name, None)` and then `exec_module` for each piketype module, in arbitrary discovery order.
- When `bar.py` is processed before `foo.py`, the line `from alpha.piketype.foo import byte_t` triggers Python's standard import machinery, which executes `foo.py` and creates **ScalarType instance #1**.
- piketype later processes `foo.py` itself: it pops `foo` from `sys.modules`, re-executes it, and produces **ScalarType instance #2**.
- `build_type_definition_map` (`src/piketype/dsl/freeze.py:92`) keys on `id(value)` and registers instance #2 under `(foo_ref, "byte_t")`.
- `bar.py`'s `__dict__` still holds instance #1.
- During `_freeze_field_type` for `bar_t.field1`, `type_definition_map.get(id(instance_1))` is `None`, so freeze falls through to `ScalarTypeSpecIR(state_kind="logic", resolved_width=8, ...)` instead of `TypeRefIR(module=foo_ref, name="byte_t")`.
- The struct field is emitted as `logic [7:0] field1`, no `TypeRefIR` exists, and the cross-module validator is never invoked.

This bug is reproducible today on `develop` and proves that the existing constitution constraint *"No cross-module type references"* is **not actually enforced** for scalar-alias references — it is silently bypassed. This spec closes that hole by **enforcing the relaxed-constraint behavior**: cross-module references work correctly, end-to-end.

### Constitution Impact

This spec relaxes the *current-milestone* constraint #4 in `.steel/constitution.md`:

> **No cross-module type references (current milestone).** Struct fields referencing types from other modules are rejected by validation. This constraint will be relaxed in a future milestone.

This is the future-milestone work referenced by that note. Landing this spec requires editing the constitution to remove the constraint and to describe the new cross-module rules. The constitution amendment is part of this feature's scope (see FR-13).

### Example

`alpha/piketype/foo.py`:

```python
from piketype.dsl import Logic

byte_t = Logic(8)
```

`alpha/piketype/bar.py`:

```python
from piketype.dsl import Struct

from alpha.piketype.foo import byte_t

bar_t = (
    Struct()
    .add_member("field1", byte_t)
    .add_member("field2", byte_t)
)
```

Expected `gen/sv/alpha/piketype/bar_pkg.sv`:

```systemverilog
// Generated by piketype
// Source: alpha/piketype/bar.py
// Do not edit by hand.

package bar_pkg;
  import foo_pkg::byte_t;

  localparam int LP_BAR_WIDTH = 16;
  localparam int LP_BAR_BYTE_COUNT = 2;

  typedef struct packed {
    byte_t field1;
    byte_t field2;
  } bar_t;

  function automatic logic [LP_BAR_WIDTH-1:0] pack_bar(bar_t a);
    return {a.field1, a.field2};
  endfunction

  function automatic bar_t unpack_bar(logic [LP_BAR_WIDTH-1:0] a);
    bar_t result;
    int unsigned offset;
    result = '0;
    offset = 0;
    result.field2 = a[offset +: 8];
    offset += 8;
    result.field1 = a[offset +: 8];
    offset += 8;
    return result;
  endfunction
endpackage
```

Today's actual output (the bug):

```systemverilog
package bar_pkg;

  ...

  typedef struct packed {
    logic [7:0] field1;
    logic [7:0] field2;
  } bar_t;
  ...
```

## User Stories

- **US-1:** As a hardware engineer, I want to define a reusable scalar alias (e.g., `byte_t`, `addr_t`) in one DSL module and reference it from struct fields in other modules, so I can share named types across a project the same way I share constants today.
- **US-2:** As a verification engineer, I want generated SystemVerilog packages to import only the cross-package symbols they actually use, so the generated `_pkg.sv` files compile against multi-package designs without name pollution and without manual `import` editing.
- **US-3:** As a firmware developer, I want generated C++ headers and Python modules that reference cross-module types to include the right `#include` / `from ... import` line, so the generated outputs build and run without hand-editing.
- **US-4:** As a tooling user, I want piketype to reject cross-module type cycles with a clear error, so I do not silently produce un-buildable SV/C++/Python.

## Functional Requirements

### FR-1: Loader — Stable Object Identity Across Modules

The loader must guarantee that, for a given DSL type defined in module `M`, every other module `N` that does `from M import T` sees **the same Python object** that piketype itself tracks for `M`.

The chosen mechanism MUST satisfy these properties:

- **Single execution per source file.** Each `*.py` DSL module is `exec`'d at most once per `piketype gen` invocation.
- **No stale references.** When piketype iterates `M.__dict__` to build the type definition map, every cross-module use of `M`'s types in any other loaded module's `__dict__` resolves to the same Python objects.
- **Deterministic load order.** Module load order is deterministic, so generated output remains byte-stable.

The implementation approach is **[NEEDS CLARIFICATION: loader strategy]** — see Open Questions Q1.

### FR-2: Freeze — TypeRefIR Carries Defining-Module Reference

`_freeze_field_type` (`src/piketype/dsl/freeze.py`) must construct a `TypeRefIR` whenever the struct member's type object is a top-level named DSL type, regardless of whether the type was defined in the same module or in another module.

- The `module` field of the resulting `TypeRefIR` is the `ModuleRefIR` of the **defining** module (where the type's `name = ...` assignment lives).
- This applies uniformly to `ScalarType`, `StructType`, `FlagsType`, and `EnumType` members.
- An anonymous (un-named) type instance from another module is impossible by construction — only named top-level types are importable from another module.

### FR-3: Freeze — `_serialized_width_from_dsl` Walks Cross-Module Refs

`_serialized_width_from_dsl` in `src/piketype/dsl/freeze.py` already recurses through DSL objects to compute serialized width for `Struct.multiple_of()`. It must continue to work when a struct member's type is a `ScalarType`/`StructType`/`FlagsType`/`EnumType` defined in another module — i.e., the function must not depend on object identity matching the current module.

No semantic change is required (the function reads the DSL object's own `width_value` / `members` / `flags` / `width`), but the test surface must include a fixture exercising cross-module width contribution under `multiple_of()`.

### FR-4: Freeze — Populate `ModuleIR.dependencies`

Today `freeze_module` (`src/piketype/dsl/freeze.py:276`) sets `dependencies=()`. After this change, freeze must populate `ModuleIR.dependencies` with one `ModuleDependencyIR(target=<ModuleRefIR of cross-module target>, kind="type_ref")` entry per **distinct** cross-module type-reference target.

- Entries are deduplicated by `(target.python_module_name, kind)`.
- Entries are sorted deterministically by `target.python_module_name` ascending.
- Same-module type references do not produce dependency entries.
- Cross-module **constant** references (already supported via `ConstRefExprIR`) also produce dependency entries, with `kind="const_ref"`. A module that has both a cross-module type ref and a cross-module const ref to the same target produces **two** entries (different `kind`).

### FR-5: Validation — Remove Cross-Module Rejection, Add Cross-Module Resolution

In `src/piketype/validate/engine.py`:

- **Remove** the rejection at lines 69-73: `"cross-module type references are not supported in this milestone"`.
- The struct-field `TypeRefIR` validation must look up the target type using the module-qualified key `(field.type_ir.module.python_module_name, field.type_ir.name)` against the existing `type_index` (which is already keyed that way at `engine.py:19-23`).
- If no target is found, raise `ValidationError` with: `"struct {struct_name} field {field_name} references unknown type {module}::{type_name}"`.
- The existing allowlist `(ScalarAliasIR, StructIR, FlagsIR, EnumIR)` for valid `TypeRefIR` targets is preserved.

### FR-6: Validation — Detect Cross-Module Struct Cycles

`_validate_struct_cycles` in `src/piketype/validate/engine.py` currently scans only same-module references (it builds the graph from `type_index` keyed by the same module). It must be extended (or a new pass added) to detect cycles **across modules**.

- The graph is `{(module_python_name, struct_name)}` nodes; edges go from a struct to every `TypeRefIR` target that resolves to a `StructIR` (regardless of module).
- A cycle anywhere in this graph raises `ValidationError` with a message that names the offending modules and types: `"recursive struct dependency detected: {mod_a}::{type_a} -> {mod_b}::{type_b} -> {mod_a}::{type_a}"`.
- Self-cycles within one module continue to be reported with the existing single-module phrasing for backward compatibility with current error-message golden expectations.

### FR-7: SystemVerilog Backend — Emit Per-Symbol `import` Statements

In `src/piketype/backends/sv/`, the synth-package emitter must emit one `import {target_basename}_pkg::{type_name};` per **distinct** cross-module type referenced by any struct field in the current module's types.

- One `import` line per unique `(target_pkg, type_name)` pair.
- Lines are sorted deterministically: first by `target_pkg` ascending, then by `type_name` ascending.
- Lines are placed inside the `package {basename}_pkg;` block, before any `localparam` declaration, separated from declarations by one blank line.
- The struct field type rendering at `src/piketype/backends/sv/view.py` `_render_sv_struct_field_type` continues to emit the **unqualified** type name (`field_type.name`) — the explicit `import` brings the symbol into scope.
- If a cross-module type appears only via a sub-struct's internal field (transitively), no import is required in the outer module's package; only **direct** references in the outer module's struct fields produce imports.

### FR-8: SystemVerilog Backend — Helper Class Cross-Module Reference

The `_helper_class_name(target.name)` rendering in `_render_sv_helper_field_decl` (`src/piketype/backends/sv/view.py:370-378`) currently emits the unqualified helper class name (e.g., `byte_ct`). For cross-module references:

- **The helper-class typedef referenced from another module must be reachable.** SystemVerilog packed-struct helper classes live in `{basename}_pkg`. The synth import (FR-7) imports the typedef but **not** the helper class wrapper.
- The synth `import` line must include both the typedef AND the helper class name: emit a wildcard `import {target_basename}_pkg::*;` instead of the per-symbol form **[NEEDS CLARIFICATION: import granularity]** — see Open Questions Q2.

### FR-9: SystemVerilog Backend — Test-Package Imports

Test packages (`{basename}_test_pkg`) currently emit `import {basename}_pkg::*;` (synth → test, same module — see `view.py:704`). When the synth package imports cross-module symbols, the corresponding test package must transitively import the cross-module test package whenever the test surface needs the cross-module helper class (e.g., for randomization, `to_slv`/`from_slv`).

- Concretely: for every unique cross-module target referenced by a struct field, emit `import {target_basename}_test_pkg::*;` in the **test** package.
- The line ordering follows the same sort rule as FR-7.

### FR-10: Python Backend — Cross-Module Imports in Generated Modules

In `src/piketype/backends/py/`, the generated `*_types.py` for a module containing cross-module references must emit:

- One `from {dotted.target.module_name}_types import {WrapperClassName}` line per distinct cross-module type, where the target module name is derived from `module.ref.python_module_name` with `_types` suffix on the basename.
  - Example: a reference to `byte_t` defined in `alpha.piketype.foo` becomes `from alpha.piketype.foo_types import byte_t_ct`.
- Sorted deterministically by full module path then class name.
- Placed in the imports section of the generated file, after stdlib imports.

### FR-11: C++ Backend — Cross-Module `#include` Directives

In `src/piketype/backends/cpp/`, the generated `*_types.hpp` for a module containing cross-module references must emit:

- One `#include "{namespace_path}/{target_basename}_types.hpp"` line per distinct cross-module type's defining module (deduplicated across multiple type refs to the same target module).
- The path matches the existing C++ output layout: `{namespace_parts joined by /}/{basename}_types.hpp`.
- Sorted deterministically.
- Placed in the include section after `<piketype_runtime.hpp>` and any other existing includes.
- The C++ field type rendering uses the namespace-qualified or unqualified form consistent with the existing namespace strategy: **[NEEDS CLARIFICATION: C++ namespacing]** — see Open Questions Q3.

### FR-12: Manifest — Surface Cross-Module Dependencies

The manifest serializer at `src/piketype/manifest/` must serialize each `ModuleIR.dependencies` entry as an object:

```json
{
  "target_module": "alpha.piketype.foo",
  "kind": "type_ref"
}
```

- The `dependencies` key in the manifest module record (currently always `[]`) is populated from `ModuleIR.dependencies`.
- Sort order matches FR-4.

### FR-13: Constitution Amendment

`.steel/constitution.md` must be updated as part of this feature's implementation:

- **Constraint #4** ("No cross-module type references") is removed.
- A new constraint is added in its place that codifies the new rules: cross-module references are allowed, must not form cycles across modules, and produce the explicit `import` / `#include` / `from ... import` lines described in FR-7 through FR-11.

The constitution edit must land in the **same merge** as the implementation, not in a follow-up.

### FR-14: Test Fixtures and Golden Files

A new fixture and corresponding goldens must be added.

- **Primary fixture** `tests/fixtures/cross_module_type_refs/`:

  - `project/alpha/piketype/bar.py` defining `bar_t = Struct().add_member("field1", byte_t).add_member("field2", byte_t).add_member("hdr", addr_t).add_member("op", cmd_t).add_member("perm", perms_t)`.
  - Goldens cover SV (`bar_pkg.sv`, `bar_test_pkg.sv`, `foo_pkg.sv`, `foo_test_pkg.sv`), Python (`bar_types.py`, `foo_types.py`), and C++ (`bar_types.hpp`, `foo_types.hpp`).
- **`multiple_of()` fixture** `tests/fixtures/cross_module_struct_multiple_of/`:

- **Negative fixture** `tests/fixtures/cross_module_cycle/`:

- **Negative fixture** `tests/fixtures/cross_module_unknown_type/`:

- **Python runtime test:** Round-trip and explicit `to_bytes()` byte-value test for a struct with a cross-module Enum and Flags member.
- **Idempotency:** Existing idempotency test infrastructure must apply unchanged — running `piketype gen` twice on the cross-module fixture produces identical output.

## Non-Functional Requirements

- **NFR-1: Backward compatibility.** Existing single-module fixtures and goldens remain byte-identical after this change. No existing golden file is rewritten by this work.
- **NFR-2: Deterministic output.** All emitted import / include / from-import lines are sorted and deduplicated; output remains byte-for-byte reproducible.
- **NFR-3: No runtime dependency growth.** Implementation uses only the existing template engine (Jinja2) and stdlib. No new third-party packages.
- **NFR-4: Performance.** Generation latency for the existing fixture suite must not regress more than 5% relative to the post-spec-010 baseline. (The `piketype gen` perf gate from spec 010 is currently open per project memory; this spec must not widen it.)
- **NFR-5: basedpyright strict.** All new Python code passes `basedpyright` strict-mode with zero errors.
- **NFR-6: Constitution alignment.** After FR-13 lands, no remaining text in `.steel/constitution.md` contradicts cross-module references.

## Out of Scope

- **Cross-module unpacked types.** Constraint #3 (packed types only) remains in force.
- **Cross-package SV `import` of constants.** Cross-module **constant** references in expressions (already partially supported by `ConstRefExprIR`) keep their current emission strategy. This spec does not change how constants are imported in SV/C++/Python; it only changes type imports. (Constant cross-module behavior is verified to still work via NFR-1.)
- **Re-exports / type aliases of cross-module types.** Defining `foo_byte_t = byte_t` in `bar.py` to "re-export" is not addressed — `foo_byte_t` would be a same-object reference; whether that should be emitted as an alias in `bar_pkg` is left for a follow-up.
- **Star imports in user DSL.** `from alpha.piketype.foo import *` is not supported by this spec — users must import each type explicitly. Discovery may or may not detect star-imported names; behavior is undefined.
- **Generated runtime cross-module ordering.** Runtime support packages (`piketype_runtime_pkg.sv`, `piketype_runtime.hpp`) are not affected.
- **Multi-repo / multi-package piketype.** Cross-module references are only resolved within a single piketype repo (single `piketype gen` invocation).

## Open Questions

- **Q1 [NEEDS CLARIFICATION]:** **Loader strategy.** Two candidate fixes for object identity (FR-1):
  - **(a) No-pop import.** Drop the `sys.modules.pop` in `python_loader.py` and rely on Python's import cache to deduplicate. Requires loading modules in topological order (or accepting that the first load wins and any later reload skips). Simple but couples piketype's lifecycle to `sys.modules` global state.
  - **(b) Two-pass load.** First pass: walk all module paths and execute each via `spec_from_file_location` + `exec_module`, **without** popping. Second pass: rebuild `__dict__` views from `sys.modules`. Same in effect as (a) but explicit.
  - **(c) Name-based identity.** Replace `id()`-keyed `type_definition_map` with a `(source.path, attribute-name-when-defined)` lookup. Requires the DSL to remember the attribute name at definition time (currently it doesn't — attribute name is recovered post-hoc by iterating `__dict__`).
  - **Recommendation:** (b) — it is the smallest change that fully removes the multi-execution pathology and is straightforward to test. Confirm with user.

- **Q2 [NEEDS CLARIFICATION]:** **SV import granularity.** Three options for the cross-module SV import:
  - **(i) Wildcard:** `import foo_pkg::*;` — matches the existing `view.py:704` style for synth-into-test imports, brings the helper class into scope as a side effect (FR-8).
  - **(ii) Per-symbol typedef only:** `import foo_pkg::byte_t;` — matches the user-supplied "expected" output literally, but does **not** bring `byte_ct` into scope; the helper-class field decl in `bar`'s helper class then has to reference `foo_pkg::byte_ct` qualified, OR add a second `import foo_pkg::byte_ct;` line.
  - **(iii) Per-symbol pair:** `import foo_pkg::byte_t; import foo_pkg::byte_ct;` — explicit and minimal; verbose for many cross-module refs.
  - **Recommendation:** (i) wildcard. It matches the existing test-package precedent, removes the helper-class scoping concern, and produces the smallest diff in `_render_sv_struct_field_type` and `_render_sv_helper_field_decl` (no qualification needed). The user's expected snippet showed `import foo_pkg::byte_t;` as illustrative; confirm whether they want literal per-symbol or are open to wildcard.

- **Q3 [NEEDS CLARIFICATION]:** **C++ namespacing.** Today the C++ backend emits types either at the global namespace or under a namespace controlled by the `--namespace` CLI flag. For cross-module type references:
  - **(i) Same namespace:** if the user-provided namespace is shared across all generated headers, then field type can be the unqualified `byte_t_ct`.
  - **(ii) Per-module namespace:** if each module's header has its own nested namespace (`alpha::piketype::foo::byte_t_ct`), the field type must be qualified.
  - **(iii) No-namespace mode:** the unqualified `byte_t_ct` resolves at link time across translation units.
  - Need to confirm the existing C++ namespace policy (single repo-level namespace? per-module?). Inspect `src/piketype/backends/cpp/` view to determine.
  - **Recommendation:** match the existing emission policy; no new namespace strategy in this spec.

- **Q4 [NEEDS CLARIFICATION]:** **Cross-module enum/flags-value name collisions.** Enum and Flags types contribute top-level identifiers (enum literals, flag bit names) into their package's symbol table. If `foo_pkg` defines `cmd_t` with a literal `IDLE`, and `bar_pkg` imports `foo_pkg::*` for `byte_t` and also defines its own `IDLE` literal in a different enum, do we report a collision at validation time? Today's `_validate_enum_literal_collision` only checks within one module.

- **Q5 [NEEDS CLARIFICATION]:** **Order of emission across modules.** Within a module, type definitions are sorted by `(source.line, name)`. Across modules, generation order is determined by discovery order. With cross-module references, should `foo_pkg.sv` always be generated before `bar_pkg.sv`? It does not affect correctness (they are separate files compiled independently), but it does affect filesystem-write order in a single `piketype gen` run, which downstream tooling occasionally cares about. Probably keep as-is unless user has a reason.
