# Gauge Review — Iteration 1

## Summary
The spec identifies the real scalar-alias bug: `python_loader.py:36` pops modules, `build_type_definition_map` keys by `id()` at `freeze.py:92-108`, and `_freeze_field_type` falls through to `ScalarTypeSpecIR` at `freeze.py:355-371`, so `engine.py:69-73` is unreachable for that case. The spec is not implementable as written because backend resolution, SV imports, C++ namespacing, loader isolation, deterministic dependency sorting, and negative conflict cases remain under-specified or wrong.

## Issues

### BLOCKING
- **FR-7/FR-8/AC-2 contradict existing SV emission and omit required imported symbols.** The spec says synth packages emit one type import (`spec.md:169-176`) and the example imports only `foo_pkg::byte_t` (`spec.md:65-77`), but existing SV templates emit `pack_<base>`, `unpack_<base>`, and `LP_<BASE>_WIDTH` for any `TypeRefIR` (`src/piketype/backends/sv/view.py:482-500`, `src/piketype/backends/sv/templates/_macros.j2:100`, `src/piketype/backends/sv/templates/_macros.j2:128-129`). A cross-module named scalar would need those symbols too unless scalar TypeRefs are deliberately special-cased, which the spec does not say. Pick one rule: wildcard import, explicit import of typedef/localparams/pack/unpack symbols, or changed scalar TypeRef pack/unpack semantics with updated expected output.
- **FR-8 is factually wrong about helper-class scope.** It says helper classes live in `{basename}_pkg` and therefore synth import must bring helper classes into scope (`spec.md:181-184`). Existing helper classes are rendered into `{basename}_test_pkg` (`src/piketype/backends/sv/templates/module_test.j2:3-16`), while synth packages contain typedefs and pack/unpack (`src/piketype/backends/sv/templates/module_synth.j2:3-19`). Move helper-class dependency handling entirely to test-package imports (FR-9) and define synth imports around synth symbols only.
- **Backends cannot resolve cross-module `TypeRefIR` with their current module-local indexes, and the spec does not require a repo-qualified backend index.** SV, Python, and C++ view builders all build `type_index` from `module.types` only (`src/piketype/backends/sv/view.py:661-662`, `src/piketype/backends/py/view.py:459`, `src/piketype/backends/cpp/view.py:604-605`) and then dereference `TypeRefIR.name` (`src/piketype/backends/sv/view.py:322-325`, `src/piketype/backends/py/view.py:201-225`, `src/piketype/backends/cpp/view.py:461-477`). Cross-module targets are not in `module.types`, so implementation will either crash or resolve the wrong local type. Add an FR requiring backend emitters/view builders to receive a repo-wide `(module_python_name, type_name)` index and use `TypeRefIR.module` everywhere.
- **C++ namespacing is unresolved even though the source code already decides the policy.** FR-11 leaves field rendering as `[NEEDS CLARIFICATION]` (`spec.md:204-210`, `spec.md:307-312`), but `_build_namespace_view` puts each generated module under its own namespace (`src/piketype/backends/cpp/view.py:241-253`) and `_build_struct_field_view` currently emits an unqualified wrapper class name (`src/piketype/backends/cpp/view.py:475-477`). An included `foo_types.hpp` will define `foo::byte_ct`/`namespace::foo::byte_ct`, not an unqualified `byte_ct` in `bar`. Specify the exact qualified C++ field type and tests for both default namespace and `--namespace`.
- **Q1 loader strategy is still open and the proposed no-pop variants ignore same-process fixture isolation.** The loader currently pops and re-executes by module name (`src/piketype/loader/python_loader.py:36-44`); tests and helper loaders repeatedly load different fixture repos that share import names such as `alpha.piketype.types` (`tests/test_view_py.py:35-45`). Dropping the pop or relying on global `sys.modules` without a per-repo session/cleanup will leak modules between fixture repos and can violate AC-7/AC-14. Resolve Q1 in the spec and require scoped module ownership/cleanup or another deterministic isolation strategy.
- **Negative unknown-type fixture is not viable as written.** FR-14 says `bar.py` should import a missing type from `foo.py` and then assert the validator's unknown-type error (`spec.md:247-248`), but Python will raise during `exec_module`, and the loader wraps it as `PikeTypeError` before freeze or validation can construct a `TypeRefIR` (`src/piketype/loader/python_loader.py:40-44`). Test FR-5 unknown-type behavior with a direct malformed IR unit test, or define a loadable DSL scenario that actually reaches validation.
- **Conflicting names are not specified, despite being in the review checklist and being correctness-critical.** The spec only leaves enum/flags literal collisions open in Q4 (`spec.md:314-315`), but unqualified SV rendering (`spec.md:176`) plus wildcard/per-symbol imports can misbind a cross-module `foo::byte_t` when `bar` also defines `byte_t`. The same problem exists for backend indexes keyed only by name. Add explicit validation or qualification rules for local-vs-imported type names, imported type-vs-type conflicts, and wildcard enum/flag literal conflicts.
- **Dependency ordering is not fully deterministic when a module has both `type_ref` and `const_ref` dependencies to the same target.** FR-4 deduplicates by `(target.python_module_name, kind)` but sorts only by `target.python_module_name` (`spec.md:145-150`), and FR-12 reuses that order for the manifest (`spec.md:223-224`). That violates constitution deterministic output requirements (`.steel/constitution.md:7`, `.steel/constitution.md:107-113`) if entries are accumulated through a set or traversal whose kind order changes. Sort by `(target.python_module_name, kind)`.
- **Open questions Q2 and Q3 leave acceptance criteria non-testable.** AC-2 accepts either wildcard or per-symbol import (`spec.md:263-264`), and AC-6 punts C++ field typing to Q3 (`spec.md:268`). Those are not acceptance criteria; they are unresolved design decisions. Resolve them before implementation so goldens have one exact byte output.

### WARNING
- **The overview misstates discovery order.** It says modules are executed in arbitrary discovery order (`spec.md:16`), but discovery returns `sorted(...)` paths (`src/piketype/discovery/scanner.py:26-34`). The real problem is dependency-unaware sorted path order plus `sys.modules.pop`, not arbitrary ordering.
- **Template-first generation is not stated as an implementation requirement.** The constitution requires meaningful generated syntax to live in templates and backend Python to prepare view models (`.steel/constitution.md:9`, `.steel/constitution.md:49-51`). The current spec requests new import/include/from-import lines but never says to add them through the existing Jinja templates/view models. Add an NFR or backend FR bullet forbidding ad hoc string concatenation for these emitted lines.
- **FR-4 expands dependencies to const refs without specifying expression traversal.** It says cross-module `ConstRefExprIR` dependencies must be emitted (`spec.md:150`), but const references can appear inside constant expressions and scalar width expressions via `_freeze_expr` (`src/piketype/dsl/freeze.py:374-408`). Specify recursive traversal over all `ExprIR` sites that contribute module dependencies.
- **Implementation staging is tightly coupled.** Once FR-2 starts preserving cross-module `TypeRefIR`, current validation rejects it (`src/piketype/validate/engine.py:69-73`) and current backends cannot resolve it. The plan should note that loader/freeze, validation, repo-wide backend resolution, and goldens have to land together or behind internal tests to preserve byte-parity at each commit.

### NOTE
- **The central bug explanation is accurate for scalar aliases.** The chain through `sys.modules.pop` (`src/piketype/loader/python_loader.py:36`), `id()`-keyed type definitions (`src/piketype/dsl/freeze.py:92-108`), scalar fall-through (`src/piketype/dsl/freeze.py:355-371`), and validator rejection (`src/piketype/validate/engine.py:69-73`) supports the Overview claim (`spec.md:9-24`).
- **All four type kinds and all three target backends are at least named.** FR-2 covers `ScalarType`, `StructType`, `FlagsType`, and `EnumType` (`spec.md:131-135`), and FR-7/FR-10/FR-11 cover SV/Python/C++ (`spec.md:169-210`), but the backend details above are not sufficient.
- **`multiple_of()` is explicitly covered.** FR-3 and AC-15 require a cross-module width/alignment fixture (`spec.md:137-142`, `spec.md:277`).

## Strengths

- The main fixture covers scalar, struct, enum, and flags references across SV, Python, and C++ goldens (`spec.md:237-242`).
- The byte-level Python runtime assertion is concrete (`spec.md:278`).
