## Memories to Save

### M-1. Project memory — namespace-flag golden orphan mapping (refinement of existing memory)

- **Type:** project (refines `project_golden_regen_pattern.md`)
- **Name:** `project_namespace_flag_orphan_mapping`
- **Content:** The existing memory `project_golden_regen_pattern.md`
  notes "namespace-flag goldens are separate orphans". This run
  pinpointed the concrete instance:
  `tests/goldens/gen/cross_module_type_refs_namespace_proj/` has **no**
  matching fixture directory. Its golden is generated by re-invoking
  the `cross_module_type_refs` fixture with the extra CLI arg
  `--namespace=proj::lib`. Source: `tests/test_gen_cross_module.py:95-107`.
  Future spec work that regenerates goldens for this fixture must use
  this `(fixture, flag_set)` pair, not look for a same-named fixture.
- **Evidence:** `specs/014-enum-pack-width-cast/tasks.md` "Plan
  corrections required" section quotes:
  ```
  $ ls -d tests/fixtures/cross_module_type_refs_namespace_proj
  ls: tests/fixtures/cross_module_type_refs_namespace_proj: No such file or directory
  ```
  And `artifacts/implementation/task2-iter1-forge.md` "Key
  Implementation Decisions" records the `--namespace=proj::lib`
  invocation.
- **Rationale:** Non-obvious because the directory naming convention
  suggests a 1-to-1 fixture-to-golden mapping. The mismatch wasted no
  time *this* run because cross-checking caught it, but a future spec
  that doesn't cross-check will silently regenerate the wrong content.
  Worth recording the specific mapping so future plans treat it as a
  known shape rather than re-discovering it.

### M-3. Project memory — venv vs system Python mismatch

- **Type:** project
- **Name:** `project_venv_required_for_piketype`
- **Content:** Running `python3 -m piketype.cli gen ...` from system
  Python fails with `ModuleNotFoundError: No module named 'jinja2'`.
  The required interpreter is `.venv/bin/python` at the project root.
  The unittest helper `_run_piketype` in `tests/test_gen_cross_module.py`

  shell session need `.venv/bin/python` explicitly.
- **Evidence:** Implementation task 2 first regen attempt (system
  python) failed; switched to `.venv/bin/python` and succeeded.
  Recorded in `task2-iter1-forge.md` "Key Implementation Decisions".
- **Rationale:** Costs ~30s of confusion per fresh-shell session.
  Pinning the path foreshortens that.

## Skill Updates

### S-1. `/steel-plan` skill — diff-base ref should not hard-code `master`

- **Skill / file:** `/steel-plan` (and propagate to `/steel-implement`,
  `/steel-validate` examples that show diff-stat commands)
- **Issue found:** The plan I wrote in iteration 1 (under the skill's
  instruction set) used `git diff --stat master ...` for verification
  gates 5 and 6. The actual base branch for this feature is
  `develop` (per `.steel/config.json:git.baseBranch`). Comparing
  against `master` includes 3 prior specs' worth of diffs and renders
  the gate useless. I caught this during T3 implementation and
  corrected to `develop`, but the plan still carries the wrong ref —
  flagged in `task3-iter1-forge.md` "Deviations from Plan" as
  "Plan amendment recommended". Without correction, the next user of
  this skill will repeat the same mistake.
- **Proposed change:** In `/steel-plan` skill, in any verification-gate
  template that mentions `git diff ... master`, replace literal
  `master` with the placeholder `<baseBranch>` and add a one-line
  instruction: "Use the value of `git.baseBranch` from
  `.steel/config.json`; do not hard-code `master` or `main`."
- **Expected impact:** Eliminates a class of plan errors that would
  otherwise be caught only at implementation T3 (the verification
  gate run). Saves one round-trip plan correction per future spec on
  any project where `baseBranch != master`.

### S-2. `/steel-implement` skill — explicit guidance on commit-prefix conflict with project Conventional Commits

- **Skill / file:** `/steel-implement` step 3 (forge commit) and
  `/steel-tasks` step T4 template
- **Issue found:** This project's Constitution lists allowed
  Conventional Commits types as `feat, fix, refactor, docs, test,
  steel`. The steel framework's commit-prefix scheme (`forge(...)`,
  `gauge(...)`, plus the listed `steel`) uses `forge` and `gauge`
  types that are NOT on the allowed list. The actual feature commit
  for this spec was `forge(implementation): tasks 1-2 iteration 1
  output`, but the Tasks document asked for `fix(sv): use width-correct
  cast in enum pack body`. Both can't be true at once. T4 gauge
  raised this as a WARNING flagged for retrospect (see below). This
  forced me to either rewrite tasks.md or accept a divergence; I
  accepted divergence and flagged it.
- **Proposed change:** Either (a) the `/steel-implement` skill should
  explicitly state "the `forge(implementation):` prefix is the carrier
  for actual code changes; if your project's Constitution requires a
  different commit type for feature work, run a final squash/amend at
  the end of the implementation stage to produce a `<type>(<scope>):`
  commit", OR (b) the workflow should auto-produce a final
  `<type>(<scope>): <description>` commit at end of implementation,
  taking the type/scope/description from a field in `tasks.md`.
  Option (b) is cleaner; option (a) is lower-risk to retrofit.
- **Expected impact:** Resolves a recurring Constitution-vs-framework
  tension. Without this, every spec on a project with strict
  Conventional Commits will leave a WARNING in the implementation
  retrospect, and the project's commit log will permanently mix
  `forge(...)` and `<type>(<scope>):` styles.

### S-3. `/steel-run-all` skill — note about clarification→planning gate

- **Skill / file:** `/steel-run-all`
- **Issue found:** The `/steel-run-all` Note says
  "Human approval gates at specification→clarification and
  planning→task_breakdown will still pause and ask the user." But
  `/steel-clarify` step 4 explicitly defines a HUMAN APPROVAL GATE at
  end of clarification (clarification→planning), and `/steel-plan`
  step 4 says "Auto-advance to `task_breakdown` stage. **No human
  approval needed here.**" The two are inconsistent: the run-all note
  lists planning→task_breakdown as a paused gate, while the plan
  skill says it's auto-advanced. During this run I followed
  `/steel-plan` step 4 (auto-advanced from planning to
  task_breakdown), which contradicts the run-all note. Either the
  note or `/steel-plan` step 4 is wrong.
- **Proposed change:** Reconcile. Either (a) update `/steel-run-all`
  Note to say "Human gate: specification→clarification ONLY; all other
  transitions auto-advance under run-all", or (b) update `/steel-plan`
  step 4 to add a human gate at planning→task_breakdown, matching the
  run-all note. Recommend (a) because adding a planning gate would
  add a stop in every workflow, which is heavier than the user's
  apparent intent (they invoked run-all to *not* be stopped at every
  gate).
- **Expected impact:** Removes ambiguity. Currently an LLM running
  `/steel-run-all` has to pick one and hope.

## Process Improvements

### PI-1. No bottlenecks (zero REVISE cycles)

Every stage converged on iteration 1. This is unusually clean. Two
plausible explanations:

1. **The bug is genuinely simple** (one-line template fix), so the
   spec, plan, and tasks all had a small surface area to disagree
   about.
2. **The Gauge may have under-pushed.** All 6 gauge reviews returned
   APPROVE on the first pass. To check whether the gauge earned its
   keep, I classified each gauge response:
   - Specification gauge: provided concrete confirmations (e.g.,
     "verified `_macros.j2` only one occurrence"). **Earned its keep**
     — independent verification, not rubber-stamp.
   - Clarification gauge: re-read iter1-forge.md and current spec.md,
     confirmed FR-3.1 list is now exhaustive. **Earned its keep.**
   - Planning gauge: produced spec coverage matrix mapping every
     FR/NFR/AC to a plan section. **Earned its keep.**
   - Task breakdown gauge: confirmed plan-correction (the
     namespace_proj fixture mismatch) was correctly flagged.
     **Earned its keep.**
   - Implementation gauges (4×): each ran independent grep / file
     reads, e.g. T2 gauge "spot-check Gates 3 and 4 by re-running
     greps and confirmed 0/8". **Earned their keep.**
   - Validation gauge: independently ran greps, located test names
     in test output, read test source to confirm `_assert_trees_equal`
     does what it claims. **Earned its keep.**

   No gauge response was a one-paragraph rubber stamp; all included
   spot-checks. Conclusion: zero-REVISE was warranted, not under-push.

### PI-4. Gemini outage at T4 — fallback to Claude was clean

Mid-implementation (T4 gauge), Gemini returned 429
`MODEL_CAPACITY_EXHAUSTED` for `gemini-3.1-pro-preview`. I switched
to Claude-as-Gauge per the skill's documented fallback ("If gauge is
`claude`: Switch to Gauge role and perform a thorough code review
yourself. Be strict.") and produced the review. The fallback worked,
but I want to flag two concerns:

1. **Self-review bias.** Claude-as-Gauge reviewing Claude-as-Forge is
   not adversarial in the same way Gemini reviewing Claude is. The
   T4 gauge artifact explicitly notes this:
   `task4-iter1-gauge.md` opens with "any conflict between this
   review and a future Gemini review should be resolved in favour of
   Gemini." This caveat is sound but currently not surfaced anywhere
   above the review file itself.
2. **No retry loop.** The skill instructs "If gauge is `gemini`: run
   `gemini ...`" but doesn't specify behaviour on 429 / capacity
   errors. I made a judgement call to fall back; the skill should
   make this explicit (retry N times, then fall back to claude with
   a flag, OR fail the stage).

**Proposed update to `/steel-implement`** (and other gauge-invoking
skills): add a paragraph "If the gauge LLM returns a transient error
(HTTP 429, capacity exhausted, network failure), retry up to 3 times
with backoff. After 3 failures, fall back to Claude-as-Gauge and mark
the gauge artifact with a `**Reviewer note:**` block stating the
fallback. Do not silently skip the gauge."
